# Диагностический отчёт по бэктесту и логам

## Контекст
- Проанализированы 3 набора сводной статистики из файла `stata` ("Первый/Второй/Третий промт").
- Проанализированы торговые логи `logs/bot.log`.
- Дополнительно проверены сырые сделки из `backtest/results/trades_*.csv` для поиска численных выбросов и противоречий полям `exit_reason/pnl/realized_r`.

## Проблема #1: Противоречие `exit_reason=stop_loss` и положительного PnL
**Лог / пример:**
- В `trades_20260314_142023.csv` у большинства сделок `exit_reason=stop_loss`, но при этом PnL часто положительный и очень большой.
- Есть сделки с `pnl=1000000.0` при `exit_reason=stop_loss`.

**Возможная причина:**
- Неверная классификация причины выхода (например, TP/таймаут/частичный выход маркируется как stop_loss).
- Пересчёт `pnl` и `realized_r` выполняется из разных источников/знаков и рассинхронизирован с финальным `exit_reason`.
- Есть вероятность пост-обработки, которая переписывает `exit_reason`, но не переписывает итоговые финансовые поля.

**Рекомендация:**
- Проверить единый источник истины для закрытия позиции: в одном месте атомарно фиксировать `exit_reason`, `exit_price`, `realized_r`, `pnl`.
- Добавить инварианты в отчёт/тесты:
  - если `exit_reason == stop_loss`, то `realized_r <= 0` и `pnl <= 0` (кроме специальных случаев breakeven, которые должны иметь отдельный `exit_reason`).
  - при любом выходе знак `realized_r` и знак `pnl` должны совпадать.

## Проблема #2: Экстремальные выбросы в excursion-метриках (`MFE/MAE`)
**Лог / пример:**
- В сводке #1: `average MFE: 86.66R`, `average MAE: 11.17R`, при медианах около 1R и ниже.
- В сводке #2: `average MFE: 70.72R`, `average MAE: 16.96R`, снова при умеренных медианах.
- В `trades_20260314_142023.csv` встречается аномальное `mae_r=3135946492530.8`.

**Возможная причина:**
- Деление на почти нулевой риск (`initial_risk -> 0`) либо ошибка нормализации в R-мультипликаторах.
- Использование некорректного `sl`/`entry` после модификаций позиции (partial/scale-in/trailing), когда risk-база меняется, а формула остаётся старой.

**Рекомендация:**
- Для всех расчётов в R ввести жёсткий floor: если `initial_risk < epsilon`, то метрика помечается как invalid и исключается из агрегатов.
- В отчёте считать среднее по winsorized-версии или дополнительно публиковать trimmed mean.
- Добавить sanity-check: `abs(mfe_r|mae_r) > 20R` логировать отдельным предупреждением с деталями сделки.

## Проблема #3: Нестабильность результатов между прогонами при одинаковых win/loss
**Лог / пример:**
- "Второй промт": 995 сделок, 428 прибыльных, суммарный PnL 368.17.
- "Третий промт": те же 995/428, но суммарный PnL уже 179.95 (почти в 2 раза ниже).

**Возможная причина:**
- Изменения в пост-обработке PnL (комиссия/проскальзывание/клиппинг), не затрагивающие исходы сделок.
- Несогласованность между полями `pnl` и `realized_r` или разный порядок применения partial TP / trailing / timeout.

**Рекомендация:**
- Фиксировать версию конфигурации расчёта PnL в каждом результирующем CSV (fees/slippage/leverage/cap rules).
- Добавить regression-check: при одинаковом наборе `entry/exit` агрегированный PnL должен быть детерминирован и повторяем.

## Проблема #4: Системные runtime-ошибки в боевых логах (NameError)
**Лог / пример:**
- `name 'MIN_VOLUME_24H' is not defined`.
- 6 повторов `name 'MIN_RR' is not defined` подряд по разным символам.

**Возможная причина:**
- Параметры риска/фильтра не импортируются в runtime-модуль или переименованы в конфиге.
- Конфигурация окружения и код не синхронизированы (часть констант ожидается в другом namespace).

**Рекомендация:**
- Пройтись по всем обязательным константам через явную валидацию на старте (fail-fast до запуска цикла по символам).
- Заменить прямые обращения к глобальным константам на централизованный объект конфигурации (`config.get(...)` с дефолтами/ошибками).

---

## Проблема #5: Повторяющийся rate-limit у биржи
**Лог / пример:**
- Серия предупреждений `Rate limit ... жду 120 секунд` + ошибка Bybit `retCode=10006 Too many visits`.

**Возможная причина:**
- Параллельный/слишком частый опрос символов без глобального rate-budget.
- После перезапусков движка повторяется burst-запросов (в логе много последовательных "Trading Engine запущен").

**Рекомендация:**
- Ввести глобальный throttler на уровень клиента биржи (token bucket / leaky bucket) и backoff c jitter.
- Сократить одновременное число символов при деградации API и кешировать данные, где возможно.

---

## Проблема #6: Неполнота данных для честного Train/Test сравнения
**Лог / пример:**
- В предоставленных артефактах нет явной разметки Train/Test/Walk-Forward folds (нет отдельных CSV/секций с OOS-метриками по фолдам).

**Возможная причина:**
- Отчёт печатает только aggregate-метрики по полному прогону без разреза IS/OOS.

**Рекомендация:**
- Для каждого фолда сохранять отдельные метрики: `fold_id`, `is_oos`, `start/end`, `trades`, `winrate`, `PF`, `avg_r`, `max_dd`, `net_pnl`.
- Отдельно считать дельту `Train vs Test` и тревожные пороги (пример: падение PF > 30%, рост DD > 50%).

## Что проверить в коде в первую очередь
1. Пайплайн закрытия позиции: порядок вычисления `exit_reason -> exit_price -> realized_r -> pnl`.
2. Формулы R-метрик и обработку случаев с `initial_risk <= 0` или близким к нулю.
3. Места, где применяются caps/клиппинг PnL (например, ограничения типа 1e6), чтобы они не маскировали реальную ошибку.
4. Инициализацию конфиг-констант (`MIN_RR`, `MIN_VOLUME_24H`) и проверку обязательных параметров на старте.
5. Контур rate-limit и поведение после рестарта движка.
